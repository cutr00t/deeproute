# DeepRoute Architecture

This document describes the system design of DeepRoute — why it exists, how it works, and the principles behind the architecture.

## Problem Statement

AI coding assistants need codebase context to be effective. The naive approach — dump all source files into the context window — doesn't scale and wastes tokens. The alternative — ask an LLM to summarize the codebase once — produces stale, inaccurate metadata that drifts from reality as code changes.

DeepRoute solves this with a **hybrid schema** that combines three complementary approaches:

| Layer | What it provides | Accuracy | Cost | Speed |
|-------|-----------------|----------|------|-------|
| AST extraction | Function names, params, return types, line numbers, classes, imports | Perfect (ground truth) | Free | Fast |
| LLM analysis | Descriptions, tags, patterns, purpose, architectural context | Approximate (interpretive) | Paid per call | Slow |
| Embeddings | Semantic similarity search ("how does X work" → finds relevant code) | Good (cosine similarity) | ~$0.0001/repo | Fast after build |

No single layer is sufficient. AST doesn't understand *why* code exists. LLMs don't reliably report *what* exists. Neither does semantic discovery well. Together they cover each other's gaps.

## Data Flow

```
Repository Source Code
        │
        ├── Scanner (file tree walk, language detection)
        │       │
        │       ├── AST Indexer ──────────────────────────┐
        │       │   Python: stdlib ast                     │
        │       │   Other langs: regex patterns            │
        │       │   Output: FunctionSpec, ClassSpec         │
        │       │   with source="ast"                      │
        │       │                                          │
        │       └── LLM Analyzer ─────────────────────────┐│
        │           Anthropic API or Vertex AI             ││
        │           Output: descriptions, tags,            ││
        │           patterns, purpose                      ││
        │           with source="llm"                      ││
        │                                                  ││
        └── Generator ─────────────────────────────────────┘│
            Merges AST (factual) + LLM (interpretive)       │
            Computes complexity scores                      │
            Writes .deeproute/v2/ JSON                      │
                    │                                       │
                    ├── Embedding Builder ──────────────────┘
                    │   OpenAI or Vertex AI embeddings
                    │   Stored as embeddings.npz
                    │
                    └── .deeproute/v2/
                        ├── manifest.json (project overview + complexity)
                        ├── modules/*.json (per-module detail)
                        ├── interfaces.json
                        ├── patterns.json
                        ├── config_files.json
                        ├── embeddings.npz
                        └── notes/*.md
```

## Schema Design

### Source Tracking

Every `FunctionSpec` and `ClassSpec` in the schema carries a `source` field:

- `"ast"` — Extracted by the AST indexer. Factually accurate: name, parameters, return type, line number, decorators, async status. No description or tags.
- `"llm"` — Generated by the LLM during analysis. Has description and tags but may have wrong line numbers, missing functions, or hallucinated classes.
- `"merged"` — AST factual data + LLM interpretive data. The ideal state: accurate location and signature with rich semantic context.

During `dr_init`, the generator produces merged entries. During `dr_update`, the AST indexer refreshes factual data without LLM calls, preserving existing descriptions and tags.

### Drift Scoring

Drift measures how much the code has changed since the schema was last updated. Computed by comparing old and new AST extractions:

```
drift = (structural_changes * 1.0 + signature_changes * 0.5) / total_symbols
```

- **Structural changes**: functions/classes added or removed (weight 1.0)
- **Signature changes**: same function name, different params or return type (weight 0.5)
- **Line-number-only changes**: ignored (cosmetic, no semantic impact)

The drift score ranges from 0.0 (no change) to 1.0 (complete divergence). When drift exceeds the threshold (default 0.3), the updater triggers LLM re-analysis for the affected module. Below threshold, only factual data refreshes — free.

This means most `dr_update` calls cost nothing. Only significant code changes trigger paid LLM analysis.

### Complexity Scoring

Each module gets a programmatic complexity score (1-10) computed entirely from AST data — no LLM needed:

| Dimension | Weight | What it measures | Low (0.0) | High (1.0) |
|-----------|--------|-----------------|-----------|------------|
| Symbol density | 15% | Functions + classes per file | 2/file | 15/file |
| API surface | 10% | Public symbols as proportion | Few | 50+ public |
| Param complexity | 15% | Avg/max params per function | 1 param | 5+ avg, 10+ max |
| Coupling | 20% | Import count + cross-module deps | 5 imports | 30+ imports, 8+ cross |
| Structural depth | 10% | Directory nesting | 2 levels | 6+ levels |
| Async ratio | 10% | Proportion of async functions | 0% | 100% |
| Volume | 10% | File count | 3 files | 30+ files |
| Method density | 10% | Methods per class (OOP) | 3/class | 12+/class |

Dimensions use sigmoid or linear normalization to 0.0-1.0, then weight and sum to produce a 1-10 score.

### Model Hints

Complexity scores derive model recommendations:

| Score | Analysis | Update | Query | Interpretation |
|-------|----------|--------|-------|----------------|
| 1-3 | haiku | haiku | haiku | Simple utility code |
| 4-5 | sonnet | haiku | haiku | Moderate complexity |
| 6-7 | sonnet | sonnet | haiku | Complex business logic |
| 8-10 | opus | sonnet | sonnet | High coupling/complexity |

`dr_plan` uses these hints to produce costed execution plans before running expensive workspace operations.

### Token Cost Model

Token estimates are based on empirical observation:

| Operation | Per file | Per symbol | Overhead | Example (20 files, 80 symbols) |
|-----------|----------|-----------|----------|-------------------------------|
| init | 100 tokens | 50 tokens | 2,000 | ~8,000 tokens |
| update | 30 tokens | 20 tokens | 1,500 | ~3,500 tokens |
| query | 20 tokens | — | 1,000 | ~1,400 tokens |

Cost per 1M input tokens: haiku $0.80, sonnet $3.00, opus $15.00.

## Incremental Updates

The updater (`updater.py`) runs a hybrid pipeline:

1. **Detect changes**: `git diff <last_sha>..HEAD` (committed) + `git diff HEAD` (uncommitted)
2. **AST re-index**: Re-extract functions/classes for changed files only (free, fast)
3. **Merge**: Update module JSON with new AST data, preserving LLM descriptions/tags
4. **Score drift**: Compare old vs new symbol tables
5. **Threshold check**: If drift >= 0.3, invoke LLM for updated descriptions/tags/patterns
6. **Update v1 layers**: If structural changes, re-analyze v1 markdown (LLM, only when needed)
7. **Update embeddings**: Rebuild search index (embedding API call, cheap)

Most updates hit steps 1-3 only — no LLM cost.

## Embedding Architecture

### Backend Abstraction

Embeddings use a separate backend from LLM analysis because Anthropic doesn't offer embedding models:

| Environment | LLM Backend | Embedding Backend | Rationale |
|-------------|-------------|-------------------|-----------|
| Personal | Anthropic API | OpenAI | Different providers for different capabilities |
| Work (GCP) | Vertex AI (Anthropic) | Vertex AI (Google) | All traffic stays in Google contract |
| Mixed | Configurable | Configurable | `DEEPROUTE_EMBEDDING_BACKEND` override |

Detection priority:
1. `DEEPROUTE_EMBEDDING_BACKEND` explicit override
2. `DEEPROUTE_BACKEND` shared setting (vertex → vertex, anthropic → openai if key exists)
3. Auto-detect from environment variables

### Storage Format

Embeddings are stored as `.deeproute/v2/embeddings.npz` — a numpy compressed archive with three arrays:

- `embeddings`: float32 matrix (N x dim), where dim is 1536 (OpenAI) or 768 (Vertex)
- `texts`: object array of the text that was embedded (for debugging)
- `metadata`: object array of JSON strings with type, name, path, module, tags

This flat-file approach requires no infrastructure (no Docker, no database). When Phase 3 adds HTTP transport, migration to ChromaDB/pgvector changes only the storage backend — the embedding generation code stays the same.

### Semantic Search

`dr_search(query="...", semantic=True)` embeds the query and computes cosine similarity against all stored embeddings. Results are filtered by a 0.1 relevance threshold and sorted by score.

This finds conceptually related code that keyword search misses:
- "model alias resolution" → `resolve_model()` at 0.576 similarity
- "how does backend detection work" → `detect_backend()` at 0.686
- "git diff changes" → `Git-Diff Driven Updates` pattern at 0.603

## Token Budget

Session-level tracking in `TokenTracker`:

- Records input/output tokens per `_call_llm` call
- Tracks breakdown by model
- Enforces budget limit as a gate before LLM calls
- Schema-mode tools always work regardless of budget

Budget state lives in server memory (resets on restart for stdio, persists across requests for HTTP). Set via `dr_config key="token_budget" value="50000"`.

## Reader Caching

`_get_reader(repo_path)` maintains a singleton cache of `SchemaReader` instances. On first access, the reader loads and parses the JSON schema files. Subsequent calls return the cached instance.

Cache is invalidated after `dr_init` or `dr_update` (which may have changed the schema files). This avoids re-parsing JSON on every `dr_lookup`/`dr_search` call within a session.

## Design Principles

1. **Factual data from code, interpretive data from LLMs**: Never trust an LLM to tell you what functions exist. Use AST for facts, LLM for meaning.

2. **Free by default, paid on demand**: Schema-mode tools cost nothing. LLM calls happen only when explicitly requested (`dr_init`, `dr_query`) or when drift crosses thresholds (`dr_update`).

3. **Graceful degradation**: No embedding key → text search works. No LLM key → schema mode works. No v2 schema → v1 markdown works. Each layer fails independently.

4. **Measure, don't guess**: Complexity scores are computed, not estimated. Drift is measured, not assumed. Token usage is tracked, not predicted.

5. **Backend-agnostic**: Same tools work on Anthropic API (personal), Vertex AI (work), or schema-only (no credentials). Embedding backend is separately configurable.
